# Phase 4：打包发布、CI/CD 与质量收口

> 周期建议：第 7-8 周  
> 阶段目标：完成跨平台打包、安装包、签名、公证、自动化流水线、系统测试和发布文档。

## 1. 实现功能

- 完成 Windows `.exe` 构建和可选 NSIS 安装包。
- 完成 macOS `.app` 构建和可选 DMG 镜像。
- 准备图标、应用元信息、版本号和构建产物命名规范。
- 完成安装、升级、卸载、首次启动、配置迁移测试。
- 完善用户文档、开发文档和故障排查文档。
- 做性能、稳定性和安全收口。

## 2. 设计方案

### 2.1 构建产物

| 平台 | 产物 | 说明 |
|------|------|------|
| Windows | `ProxyServer.exe` | 单文件可执行程序 |
| Windows | `ProxyServer-Setup-x.y.z.exe` | NSIS 安装包，支持快捷方式和卸载 |
| macOS | `ProxyServer.app` | App Bundle |
| macOS | `ProxyServer-x.y.z.dmg` | DMG 分发镜像 |

版本号采用语义化版本：`MAJOR.MINOR.PATCH`。构建产物必须包含版本号、平台和架构。

### 2.2 CI/CD

推荐 GitHub Actions 矩阵：

- Windows amd64。
- macOS amd64。
- macOS arm64。

流水线阶段：

1. Checkout。
2. 安装 Go 和 Node.js。
3. 安装 Wails CLI。
4. 安装前端依赖。
5. 执行 Go 测试。
6. 执行前端 lint 和 build。
7. 执行 Wails build。
8. 签名和打包。
9. 上传构建产物。

签名密钥、证书和 Apple 公证凭据必须通过 CI secret 注入，禁止提交到仓库。

### 2.3 代码签名

Windows：

- 使用 EV 或 OV 代码签名证书。
- 对 `.exe` 和安装包分别签名。
- 安装包可配置防火墙例外提示，但不得静默放开未经用户确认的高风险规则。

macOS：

- 使用 Apple Developer ID Application 证书签名。
- 对 `.app` 完成 notarize。
- stapler 公证票据后再生成或发布 DMG。

### 2.4 测试收口

测试范围：

- 单元测试：配置、协议解析、relay、认证、统计。
- 集成测试：SOCKS5、HTTP CONNECT、服务启停、端口冲突。
- UI 验证：核心页面渲染、配置保存、日志展示、图表刷新。
- 打包验证：安装、启动、退出、卸载、升级。
- 长稳测试：持续代理连接、频繁启停、日志轮转。

性能基线建议：

- 1000 并发连接下服务保持可响应。
- 大文件传输过程中内存稳定，无持续增长。
- UI 日志列表在持续写入时不明显卡顿。

### 2.5 发布文档

必须包含：

- 用户安装与首次启动说明。
- 代理客户端配置示例。
- 常见错误：端口占用、认证失败、系统拦截、WebView2 缺失。
- 配置文件路径说明。
- 日志文件路径说明。
- 隐私与安全说明。

## 4. 验收标准

- CI 能在 Windows 和 macOS runner 上产出构建包。
- Windows 可执行文件和安装包可以正常启动、安装和卸载。
- macOS `.app` 可以正常启动，完成签名和公证后不触发 Gatekeeper 阻断。
- 所有自动化测试通过。
- 发布包包含明确版本号和平台架构信息。
- 用户文档覆盖安装、配置、日志、常见错误和卸载。
- 发布前检查表全部完成。

## 5. 风险与处理

| 风险 | 处理方案 |
|------|----------|
| macOS 公证失败 | 单独记录签名、公证、stapler 日志，CI 失败时保留中间产物 |
| Windows SmartScreen 拦截 | 使用可信代码签名证书并保持稳定发布渠道 |
| CI 环境与本地构建不一致 | 固定 Go、Node、Wails 版本，构建脚本进入仓库 |
| 发布包遗漏运行资源 | 打包后执行安装和首次启动冒烟测试 |
| 许可证合规风险 | 发布前生成依赖清单并检查第三方许可证 |
